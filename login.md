# 사용자 인증 및 멀티테넌시 설계 노트

README의 **향후 확장 아이디어** 중「사용자·인증 및 멀티테넌시」를 이 프로젝트(Express + Prisma + PostgreSQL + React)에 맞춰 구체화한 문서입니다. 구현 전 검토·우선순위 정리용입니다.

---

## 현재 상태

- API는 **인증 없이** 동작하며, `Chunk`·`Question`·`WrongAnswer`·`ExamAttempt` 등 모든 데이터가 **전역(단일 테넌트)** 으로 쌓입니다.
- LLM 모델·API 키는 **서버 프로세스 메모리**에만 있어 사용자/조직 단위 구분이 없습니다.

---

## 목표를 나누기

| 구분 | 의미 |
|------|------|
| **인증(Authentication)** | 누가 요청했는지 확인 (로그인 세션 또는 토큰). |
| **인가(Authorization)** | 해당 사용자가 이 리소스(소스·문제·시험 이력)에 접근할 수 있는지 판단. |
| **멀티테넌시** | A 사용자(또는 A 조직)의 PDF·청크·문제·오답이 B와 **논리적으로 격리**되는 것. |

실무에서는 **「사용자 = 테넌트」**(B2C)로 시작하거나, **「조직 = 테넌트, 사용자는 조직 소속」**(B2B)로 나눕니다. 후자는 팀/학원 단위 공유가 필요할 때 선택합니다.

---

## 멀티테넌시 모델 권장

### 1) 행 단위 격리(권장, 이 스택에 적합)

모든 도메인 테이블에 `tenantId`(또는 `userId`만 쓰는 B2C)를 두고, **모든 쿼리·생성에 조건을 강제**합니다.

- **장점**: 단일 DB·단일 스키마, Prisma 마이그레이션 단순, 운영 비용 낮음.
- **주의**: 개발자 실수로 `where` 누락 시 데이터 유출 → **미들웨어 + Repository/서비스 레이어**에서 테넌트 ID를 주입하는 패턴이 안전합니다.

### 2) 스키마/DB 분리

테넌트별 PostgreSQL 스키마 또는 DB 인스턴스.

- **장점**: 강한 격리, 대형 엔터프라이즈 규모에서 간혹 사용.
- **단점**: 마이그레이션·백업·연결이 복잡해져 **본 프로젝트 규모에는 과한 경우가 많음**.

**결론**: 초기에는 **행 단위 + `tenantId`(또는 `userId`)** 로 가는 것을 권장합니다.

---

## 데이터 모델 확장 방향 (Prisma)

현재 `Chunk`, `Question`, `WrongAnswer`, `ExamAttempt`(및 `ExamAttemptItem`)에 소유 주체가 없습니다. 확장 시 예시는 다음과 같습니다.

1. **`Tenant`** (선택): `id`, `name`, `plan` 등 — B2B 또는 이후 팀 기능 대비.
2. **`User`**: `id`, `email`, `passwordHash` 또는 외부 IdP 식별자, `tenantId`(또는 1:1이면 `tenant` 없이 `User`만).
3. **기존 모델에 FK 추가**: 예) `Chunk.tenantId` + `userId`(업로더), `Question`/`WrongAnswer`/`ExamAttempt`에도 동일 계열.

인덱스: `(tenantId, source)`, `(tenantId, createdAt)` 등 조회 패턴에 맞춰 추가.

RAG 검색·문제 생성 시에는 **반드시** `tenantId`(및 필요 시 `userId`)로 청크를 제한해, 다른 테넌트 텍스트가 컨텍스트에 섞이지 않게 합니다.

---

## 인증 방식 선택

| 방식 | 적합한 경우 |
|------|-------------|
| **세션 + HttpOnly 쿠키** | 같은 도메인(또는 서브도메인)에 프론트·API를 두고, CSRF 대책을 같이 설계할 때. |
| **JWT(액세스·리프레시)** | 모바일·별도 API 클라이언트, 또는 완전 분리된 SPA 도메인에서 Bearer 헤더로 통일할 때. |
| **OAuth 2.0 / OpenID Connect** (Google, GitHub 등) | 비밀번호 관리 부담을 줄이고 가입 전환율을 올리고 싶을 때. 비밀번호 로그인은 나중에 추가 가능. |

Express에서는 `passport` 또는 `better-auth` 같은 라이브러리, 또는 **자체 구현 JWT + refresh 로테이션** 중 팀 역량에 맞게 선택하면 됩니다.

## API·서버 측 필수 패턴

1. **인증 미들웨어**: 요청마다 `req.user` / `req.tenantId` 설정. 실패 시 `401`.
2. **인가**: 리소스 id로 조회할 때 `where: { id, tenantId }` 형태로 **항상** 스코프.
3. **업로드·생성**: body에 테넌트를 넣지 말고, **토큰/세션에서만** `tenantId` 결정 (클라이언트 조작 방지).
4. **SSE(`/api/tutor`)**: 쿼리스트링·헤더의 토큰 또는 쿠키 기반으로 동일하게 인증 (EventSource는 커스텀 헤더 제한이 있어 쿠키 또는 쿼리 파라미터 설계 필요).

## LLM 설정과 멀티테넌시

현재는 서버 전역 메모리입니다. 멀티테넌시 도입 시 선택지는 다음과 같습니다.

- **테넌트(또는 사용자)별 설정 테이블**에 `provider`, `model`, 암호화된 API 키 저장.
- 또는 **환경변수는 플랫폼 기본값**만, 유료 키는 사용자가 UI에서 넣은 값을 자신의 행에만 저장.

키는 평문 컬럼 저장을 피하고, **KMS 또는 앱 레벨 암호화**(예: 마스터 키로 AES)를 검토합니다.

---

## 프론트엔드(React)

- 로그인/회원가입 페이지, 보호 라우트(`React Router`에서 미인증 시 `/login` 리다이렉트).
- `api.js` fetch에 `credentials: 'include'`(쿠키 세션) 또는 `Authorization: Bearer …`(JWT) 일관 적용.
- 기존 `VITE_API_URL` 구조는 유지 가능.

---

## 보안 체크리스트 (최소)

- 비밀번호: 해시는 **bcrypt/argon2** 등 전용 알고리즘.
- 쿠키: `Secure`, `SameSite`, 프로덕션에서 `HttpOnly`.
- 레이트 리밋: README에 있는 Redis 확장 아이디어와 결합해 **IP + userId** 기준 제한.
- 감사: 로그인 실패·민감 설정 변경 로그(선택).

---

## 도입 순서 제안 (마이그레이션)

1. Prisma에 `User`(및 필요 시 `Tenant`) 추가, 기존 테이블에 nullable `tenantId` 도입 → 기존 행은 시스템 테넌트 또는 마이그레이션 스크립트로 일괄 할당.
2. 인증 API·미들웨어만 붙이고, 읽기 API부터 `tenantId` 필터 적용.
3. 쓰기·업로드·RAG·시험·오답 경로 순으로 확대.
4. LLM 설정을 DB(테넌트 단위)로 이전.
5. `tenantId`를 NOT NULL로 전환 및 테스트.

---

## 요약

- **멀티테넌시**는 이 코드베이스에는 **행 단위 `tenantId`/`userId` + 모든 쿼리 스코프**가 현실적입니다.
- **인증**은 세션 쿠키 또는 JWT + OAuth 중 하나를 선택해 Express 미들웨어와 Prisma 쿼리를 일관되게 묶는 것이 핵심입니다.
- RAG·PDF·LLM 키까지 테넌트 경계를 넣지 않으면 **데이터·프롬프트 유출** 위험이 있으므로, 설계 단계에서 함께 잡는 것이 좋습니다.

이 문서는 구현 명세가 아니라 **방향 합의용**이며, 실제 컬럼명·인덱스·라이브러리는 팀 표준에 맞게 조정하면 됩니다.
