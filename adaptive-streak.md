# 적응형 난이도·학습 스트릭 대시보드 설계 노트

README **향후 확장 아이디어** 중「적응형 난이도·학습 스트릭 대시보드」를 이 코드베이스(퀴즈·시험·오답·토픽·난이도 문자열)에 맞춰 구체화한 문서입니다. 구현 전 범위·지표 정의용입니다.

---

## 현재 상태 (활용 가능한 데이터)

| 영역 | 이미 있는 것 |
|------|----------------|
| **문항** | `Question`에 `topic`, `difficulty`(프롬프트 기준 `상\|중\|하`, 기본값 `중`) |
| **시험 이력** | `ExamAttempt` / `ExamAttemptItem` — 회차별 정답률·토픽·소스 스냅샷 |
| **오답** | `WrongAnswer` — `reviewCount`, `nextReview`(복습 주기 힌트로 쓸 수 있음) |
| **약점 시각화** | 클라이언트 `weakness.js` + `WeaknessChart` — **세션 내** 토픽별 정답률 |

아직 **일별 집계 테이블**, **사용자별 히스토리**(로그인 없음), **난이도 자동 조절 로직**은 없습니다. 멀티유저·인증이 생기면 대부분의 지표는 **`userId`(또는 `tenantId`)로 스코프**하는 것이 전제가 됩니다.

## 1. 적응형 난이도 (Adaptive Difficulty)

### 목표

같은 토픽·소스 안에서 **너무 쉬운 문제만 반복하지 않고**, **한 번에 너무 어렵지도 않게** 출제 난이도를 맞추는 것.

### 난이도 표현

- **단계 유지**: 지금처럼 LLM에게 `상|중|하` 문자열로 지시 (변경 최소).
- **확장**: 숫자 스케일(1–5)로 내부만 관리하고 생성 시 문자열로 매핑해도 됨.

### 신호(입력)로 쓸 만한 것

- 최근 **N문항** 또는 **최근 K일** 동안 해당 `topic`(또는 `source+topic`)의 **정답률**.
- **연속 정답 / 연속 오답** 스트릭(문항 단위).
- 시험 모드 `ExamAttemptItem.isCorrect` 집계.

### 조절 로직 예시 (단순 휴리스틱)

- 최근 10문항 정답률 **≥ 80%** → 다음 생성 시 난이도 한 단계 **상향**(중→상).
- **≤ 40%** → **하향**(상→중, 중→하).
- 그 사이 → 유지.

고급: 항목 반응 이론(IRT)·지식 추적(Knowledge Tracing)은 데이터·표본이 쌓인 뒤 검토.

### 구현이 닿는 지점

- `POST /api/questions/generate` 요청 전·후에 **서버가 `difficulty` 후보를 계산**해 `rag.generateQuestions`에 넘기거나, 프롬프트의 출제 지시에 반영.
- 토픽별 이력이 필요하면 **`QuestionAttempt` 같은 이벤트 테이블**(문항 id, 정오답, 시각, userId)을 두는 편이 깔끔함. 현재는 퀴즈 세션이 DB에 남지 않을 수 있어 **시험·오답 저장분만**으로는 표본이 편향될 수 있음.

## 2. 학습 스트릭 (Streak)

### 목표

**매일 꾸준히 학습했다는 동기**를 주기 위해, 연속 학습 일수·캘린더 UI를 대시보드에 노출.

### 반드시 정의할 것

1. **“오늘 학습 완료” 조건**  
   예: 문제 **최소 1문제 풀이**, 시험 **1회 제출**, **복습 N분** 등. 조건이 애매하면 스트릭이 깨지기 쉽거나 반대로 너무 쉬워짐.
2. **타임존**  
   `Asia/Seoul` 고정 vs 브라우저/프로필 설정.
3. **시각 기준**  
   자정 리셋인지, 롤링 24시간인지(후자는 스트릭 UX에 잘 안 씀).

### 데이터

- **`StudyDay` / `DailyActivity`**: `(userId, date)` 유니크, 그날 충족한 활동 플래그.
- 또는 **이벤트 로그**만 쌓고 일별로 집계(배치 또는 조회 시).

스트릭 계산: 연속된 `date` 구간의 최대 길이 또는 **현재 진행 중 스트릭**(어제 또는 오늘 활동이 있어야 유지).

### 엣지 케이스

- 여행·병결용 **“스트릭 보존권”**(주 1회 면제) — 선택 사항.
- 로그인 전 **로컬 스트릭** vs 로그인 후 **서버 동기화** 충돌 처리.

## 3. 대시보드 (UI·API)

### 위젯 아이디어

| 위젯 | 데이터 소스(개략) |
|------|-------------------|
| **오늘의 스트릭 / 최장 기록** | 일별 활동 테이블 또는 집계 API |
| **최근 7일·30일 활동 히트맵** | GitHub 잔디 스타일 |
| **토픽별 숙련도** | 시험·(추가 시) 퀴즈 이벤트로 정답률·추세 |
| **적응형 난이도 상태** | 토픽별 “현재 권장 난이도” 배지 |
| **복습 큐** | `WrongAnswer.nextReview`가 오늘 이전인 항목 수 |

### API 예시 (향후)

- `GET /api/stats/streak` — 현재 스트릭, 최장, 이번 주 요약.
- `GET /api/stats/topics` — 토픽별 정답률·추천 난이도(서버 계산).
- `POST /api/activity/ping` — 하루 1회 조건 충족 기록(스팸 방지·idempotent).

프론트는 기존 **대시보드 페이지**에 섹션을 추가하거나, `/dashboard/insights` 같은 하위 라우트로 분리.

---

## 4. Redis·배치와의 관계

README의 **Redis**(시험 세션·레이트 리밋)와 겹치지 않게 역할을 나누면 좋습니다.

- **스트릭·일별 카운트**: 장기적으로는 **Postgres가 소스 오브 트루스**, Redis는 캐시·오늘 조건 중복 방지(idempotency)에 선택적 사용.
- **집계가 무거워지면** 일별 배치로 `topic_stats` 테이블 갱신.

---

## 5. 도입 순서 제안

1. **이벤트 로그** 최소 스키마(또는 시험·오답만으로 1차 지표) + 토픽별 정답률 API.
2. **난이도 휴리스틱**을 `questions/generate` 경로에만 연결해 A/B 느낌으로 검증.
3. **일별 활동 + 스트릭** 규칙 확정 후 DB·대시보드 위젯.
4. 인증 도입 시 모든 집계에 **userId** 스코프 적용([login.md](./login.md)와 정합).

---

## 요약

- **적응형 난이도**는 토픽(·소스)별 **최근 정답률**로 `상/중/하`를 조절하는 것부터 시작하기 좋고, 정확도를 올리려면 **문항 단위 시도 기록**이 유리합니다.
- **스트릭**은 **“하루 1회 학습” 정의**와 **타임존**을 먼저 고정해야 구현·UX가 흔들리지 않습니다.
- **대시보드**는 기존 대시보드·약점 차트를 확장해 스트릭·토픽 숙련도·복습 큐를 한 화면에 묶는 형태가 자연스럽습니다.

구현 명세가 아니라 **범위·지표·데이터 공백**을 메우기 위한 노트입니다. 실제 컬럼·엔드포인트명은 구현 시 조정하면 됩니다.
